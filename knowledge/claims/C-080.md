Claim ID:        C-080
Type:            OBLIGATION + ARCHITECTURAL_CONSTRAINT
Statement:       All automated tests on the WAOOAW platform — unit tests, integration tests, Constitutional Compliance Tests (CCTs), performance tests, security tests, and pipeline tests — MUST execute inside Docker containers. Virtual environments (venv, virtualenv, pyenv, conda, pipenv) are PROHIBITED as the execution environment for any test on this platform. This mandate applies without exception to: local developer machines, GitHub Actions CI runners, Codespace environments, and any future compute environment. The only compliant test execution command is `docker compose run --rm test-runner <test-command>` or an equivalent Docker invocation. A test that was verified only in a virtual environment has NOT been verified.
Confidence:      100%
Source:          C-071 (Quality Obligation — quality is constitutional); C-065 (SDLC Separation of Duties — test environment must be reproducible and not dependent on developer machine state); C-072 (Coding Standards Obligation — standards apply to test code and its execution environment); GENESIS Engineering Quality Mandate ("A standard without enforcement is a suggestion — every standard here has an automated enforcement mechanism").
Depends On:      C-071 (Quality Framework), C-072 (Coding Standards), C-065 (SDLC Separation)
Produces:        Dockerfile.test-runner (single source of truth for test execution environment); docker-compose.yml test-runner service; engineering-standards.md §0.3 Docker pre-flight commands; QA-STRATEGY.md §Docker Execution Mandate; CCT-PIPE-01 updated (verify conftest.py not using venv imports)

Reviewer Notes:  WHY VIRTUAL ENVIRONMENTS ARE PROHIBITED:

                 A virtual environment is a developer-local construct. Its state depends on:
                 - The developer's machine OS and Python version
                 - The order packages were installed
                 - Whether the developer remembered to activate it
                 - Whether conflicting global packages exist

                 A test that passes in a venv on machine A may fail on machine B.
                 A test that passes in a venv may fail in CI.
                 A test that was verified in a venv was NOT constitutionally verified.

                 Docker containers remove ALL of this variability:
                 - Same OS, same Python version, same package versions
                 - Same execution on every machine, every CI run, every time
                 - The Dockerfile IS the specification of the test environment
                 - "It worked on my machine" is constitutionally inadmissible

                 WHAT COUNTS AS DOCKER-COMPLIANT:
                 - `docker compose run --rm test-runner pytest tests/`            ✅
                 - `docker run --rm waooaw-test-runner pytest tests/`             ✅
                 - GitHub Actions ubuntu-latest runner running docker commands    ✅
                 - testcontainers (already mandated in QA-STRATEGY.md)            ✅

                 WHAT IS PROHIBITED:
                 - `source .venv/bin/activate && python3 -m pytest`               ❌
                 - `pip install && pytest`                                         ❌
                 - `python3 -m pytest` on host system (no Docker)                 ❌
                 - `poetry run pytest`                                             ❌
                 - Any pyenv/conda/virtualenv invocation for test execution       ❌

                 NOTE: pyproject.toml, ruff.toml, mypy.ini remain as configuration
                 files for tools. They are read BY the Docker container. Having
                 them in the repo does not violate C-080 — running them outside
                 Docker does.

                 GITHUB ACTIONS INTERPRETATION:
                 GitHub Actions ubuntu-latest runners are ephemeral, isolated VMs.
                 Pipeline tooling (scripts/*.py runner/reviewer scripts) may use
                 the system Python in Actions runners — these are CI execution scripts,
                 not tests. However, ALL test invocations in Actions workflows must
                 use `docker compose run --rm test-runner` or equivalent.
