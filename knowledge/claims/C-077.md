Reviewer Notes:  CEILING RATIONALE:
                 ₹5,000/month is generous relative to the zero-cost architecture targets.
                 The actual expected spend with FinOps patterns in place is ₹200-₹800/month.
                 The ₹5,000 ceiling provides 6-25× headroom — it is a safety fence, not an operating target.

                 WHAT COUNTS TOWARD THE CEILING:
                 - GitHub Actions minutes for autonomous-sprint.yaml (Claude Sonnet API calls)
                 - Preflight job LLM calls (model_hint: auto → Gemini Flash or LOCAL)
                 - Spec validation LLM calls in SPEC phase
                 - Any ADR-030 zero-cost dev agent LLM calls when IB-020 is implemented

                 WHAT DOES NOT COUNT:
                 - Customer-facing agent LLM costs (governed by C-043 and ADR-024)
                 - Steward Assistant sessions (governed by ADR-028 — always FRONTIER)
                 - CI/CD infrastructure costs (covered by C-067 environment ceilings)
                 - GitHub Actions runner minutes for non-LLM jobs (linting, CCTs, builds)

                 ENFORCEMENT MECHANISM:
                 autonomous-sprint.yaml preflight job reads institutional.development_token_usage
                 for the current calendar month, converts to INR at current provider rates,
                 compares to ₹5,000. If projected month-end spend > ₹5,000: HALT + alert.
                 If current month spend > ₹4,000 (80%): WARNING in preflight output only.

                 This claim also authorizes IB-020 (ADR-030) to proceed — the budget ceiling
                 is now defined, which was the prerequisite for the zero-cost dev agent design.

                 AMENDMENT — LLM OUTPUT TOKEN FLOOR (2026-07-23):
                 Evidence from WC-012 runs #31/#32 established that a max_tokens ceiling of
                 8,000 output tokens causes output truncation for scaffold tasks: the agent
                 generates incomplete files, validation fails, retries waste budget, and the
                 task never succeeds. The floor below is therefore COST-REDUCING, not cost-
                 increasing — it prevents wasteful retry loops.

                 TOKEN OUTPUT FLOOR (governs autonomous-sprint.yaml code generation tasks):
                 - Minimum max_tokens for scaffold tasks (WC-0XX-01): 16,000
                 - Minimum max_tokens for implementation tasks (WC-0XX-02 to 04): 8,000
                 - Maximum max_tokens for any single call: 32,000 (cost control ceiling)

                 Rationale: A scaffold task generates .csproj + proto + service class + startup =
                 ~12,000–20,000 output tokens for .NET. A Next.js scaffold is ~25,000 tokens.
                 Setting max_tokens below the natural output size causes mid-response truncation
                 that is indistinguishable from success at the API level — the response returns
                 200 OK but the generated code is incomplete. This is a token budget trap.

                 MOBILE EXTENSION NOTE:
                 Flutter/Dart mobile app scaffolds are estimated at 30,000–40,000 output tokens
                 (app structure + navigation + auth + constitutional integration). When a mobile
                 sprint is authorized, max_tokens for WC-0XX-01 mobile scaffold MUST be set to
                 32,000 (the ceiling). If the scaffold cannot be generated within 32,000 tokens,
                 the sprint task MUST be split into smaller sub-tasks before execution.
