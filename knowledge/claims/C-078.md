Claim ID:        C-078
Type:            OBLIGATION + ARCHITECTURAL_CONSTRAINT
Statement:       Every prompt sent to an external LLM provider (Google Vertex AI, Azure OpenAI, Sarvam AI, or any future provider governed by ADR-029) must pass through a PII Scrubbing Layer before dispatch. This layer detects and replaces personally identifiable information (PII) with stable, reversible placeholder tokens before the prompt leaves the WAOOAW network boundary. PII must never travel to an external LLM provider in its original form. This obligation is non-waivable: it applies to all inference tiers (LOCAL providers running on-premise are exempt), all agent types, all prompt categories, and all environments (dev, QA, UAT, prod).
Confidence:      100%

Reviewer Notes:  WHY THIS IS CONSTITUTIONAL AND NOT JUST TECHNICAL:

                 WAOOAW's customers (Dr. Mehta, Suresh, Rahul, Sana) provide their personal
                 information to WAOOAW in trust. They did not consent to have their phone numbers,
                 business names, addresses, Aadhaar numbers, or PAN numbers transmitted to
                 Google, Microsoft, or any other external AI company.

                 Sending raw PII to external LLM providers:
                 1. Violates DPDPA Section 8(7) — no technical safeguard
                 2. Violates C-048 — the platform is using customer data beyond stated purpose
                 3. Undermines constitutional trust (C-002) — trust is built by evidence of protection,
                    not just by claims of protection

                 The PII Scrubbing Layer must operate as follows:
                 - Detect: AI4Bharat IndicNER (Indian-language-aware) + structural regex for phone/email/Aadhaar/PAN
                 - Replace: deterministic tokens ({{CUSTOMER_PHONE}}, {{CUSTOMER_NAME}}, etc.)
                 - In-flight map: token → original value held in-memory for response reconstruction only
                 - Reasoning traces: stored with tokens, not original values (WAOOAW IP, never needs PII)
                 - Evidence records: stored with tokens; original values in customer-controlled Tier 2 store only

                 LOCAL-tier (Ollama, AI4Bharat on-premise) is exempt because the data never leaves
                 the WAOOAW infrastructure boundary. This exemption must be enforced by architecture
                 (no external network call), not by configuration flag.

                 PII detection False Negatives (PII that slips through):
                 - Acceptable: < 0.1% false negative rate (target for IndicNER + regex combined)
                 - Unacceptable: any structural PII (phone, email, Aadhaar format) that slips through
                 - CCT-PII-02 must include adversarial inputs testing detection limits

                 Implementation authority: AI Runtime — Component 7 (PII Scrubber).
                 See architecture/reference/pii-masking-pipeline.md for full specification.
