Claim ID:        C-066
Type:            PLATFORM_INTEGRITY + AUTONOMOUS_GOVERNANCE
Statement:       Autonomous software development work by WAOOAW AI Agents is governed by a four-tier authorisation model. The tier determines whether a WAOOAW AI Agent may begin implementation work autonomously or must wait for human approval. Tier assignment is determined by the nature and risk of the work, not by the agent's confidence. No agent may self-assign a lower tier to begin work sooner. The three human stewards (C-064) retain veto authority at any tier and at any stage via Human Override (C-001).
Confidence:      99%
Source:          C-001 (Human Override — unconditional), C-064 (Three-Human Institution — Yogesh, Sujay, Ojal govern; AI agents execute within their licensed authority), Constitution Article III (Second Law — authority is continuously licensed, not assumed), C-065 (SDLC Separation of Duties — tiers govern who approves spec before authorship begins).
Depends On:      C-001, C-064, C-065
Produces:        GitHub Issue label convention for tier assignment; WAOOAW AI Agent — Platform IT Expert Decision Space; autonomous development workflow
Contradicted By: NONE
Status:          RATIFIED — 2026-07-18 (Founder authorization)
Reviewer Notes:  THE FOUR AUTHORISATION TIERS:

                 TIER 0 — EMERGENCY RESPONSE (Fully Autonomous, Immediate)
                 ─────────────────────────────────────────────────────────
                 Trigger:    Production incident (service down, constitutional violation,
                             security breach, Emergency Stop failure)
                 Agent action: Implement fix immediately, notify all three humans,
                               create post-incident report
                 Approval:   NOT required before action — speed is constitutional
                 Human role: Notified immediately; may override/roll back at any time
                 GitHub:     Label `tier:0-emergency` auto-applied by monitoring alert
                 Evidence:   CE.RecordEvidence(EMERGENCY_FIX_STARTED) before first commit
                 Example:    Constitutional Engine is down; no customer can hire an agent

                 TIER 1 — BUG FIX (Autonomous within approved spec)
                 ─────────────────────────────────────────────────────
                 Trigger:    Bug report, failing CCT, test failure
                 Agent action: Create spec as GitHub Issue comment; implement on approval
                 Approval:   Sujay approves by applying label `approved:sujay`
                 Human role: Sujay reviews spec (5–10 minutes); can modify or reject
                 GitHub:     Label `tier:1-bugfix`; blocks implementation until approved label
                 Evidence:   CE.RecordEvidence(BUGFIX_SPEC_APPROVED) before first commit
                 SLA:        Sujay responds within 4 working hours
                 Example:    WhatsApp OTP delivery fails for 3 users; fix in OTP service

                 TIER 2 — FEATURE / ENHANCEMENT (Approved IB item)
                 ──────────────────────────────────────────────────
                 Trigger:    Approved IB item (status: authorized in INSTITUTIONAL_BACKLOG.md)
                 Agent action: Pick up the IB item; create detailed implementation spec;
                               begin only after both spec approval AND IB authorization
                 Approval:   Sujay approves spec via `approved:sujay`; IB item must have
                             `status:authorized` in backlog
                 Human role: Sujay approves spec; Yogesh has already approved IB item
                 GitHub:     Label `tier:2-feature`; branch `ib/{IB-number}/{slug}`
                 Evidence:   CE.RecordEvidence(FEATURE_IMPLEMENTATION_STARTED) before commit
                 Example:    IB-009 Foundation Implementation sprint

                 TIER 3 — CONSTITUTIONAL / ARCHITECTURAL CHANGE
                 ──────────────────────────────────────────────
                 Trigger:    Change to constitution/, knowledge/claims/, adr/, agent specs
                 Agent action: Create detailed spec and constitutional impact assessment
                 Approval:   Yogesh (Founder) must explicitly approve via PR comment
                             "/approved" (per existing GitHub operating model)
                 Human role: Yogesh reviews the constitutional impact; full Founder Gate
                 GitHub:     Label `tier:3-constitutional`; @dlai-sd required in CODEOWNERS
                             (already enforced); no merge without Yogesh approval
                 Evidence:   CE.RecordEvidence(CONSTITUTIONAL_CHANGE_APPROVED) before merge
                 Example:    Adding a new constitutional claim; amending an ADR;
                             changing the Emergency Stop architecture

                 WHAT CAN NEVER BE CHANGED AUTONOMOUSLY (beyond Tier 3):
                 - constitution/CONSTITUTION.md (Class 1 — Immutable)
                 - constitution/GENESIS.md (Class 1 — Immutable)
                 - Any record in the Constitutional Audit Ledger (C-007 — append-only)
                 - GitHub branch protection rules (would allow self-merge — C-065 violation)
                 - CODEOWNERS file (would remove Founder review — C-065 violation)

                 LABEL CONVENTION (GitHub Issue labels):
                 `tier:0-emergency`     — Tier 0
                 `tier:1-bugfix`        — Tier 1
                 `tier:2-feature`       — Tier 2
                 `tier:3-constitutional` — Tier 3
                 `approved:sujay`       — Sujay has approved spec (Tier 1 and 2)
                 `approved:yogesh`      — Yogesh (Founder) has approved (Tier 3)
                 `status:implementing`  — Agent is actively implementing
                 `status:pr-open`       — PR created, awaiting review
                 `status:deployed`      — Successfully deployed
                 `status:blocked`       — Blocked — needs human input
