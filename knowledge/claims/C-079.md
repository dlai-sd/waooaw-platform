Claim ID:        C-079
Type:            ARCHITECTURAL_CONSTRAINT + OBLIGATION
Statement:       When the Constitutional Engine (CE) is unavailable — due to network failure, service crash, pod restart, or any other cause — all platform operations that require CE authorization must enter a Fail-Safe Halt state immediately. No PAAS session may execute a new action, no approval-gate action may transition to EXECUTED, and no new authority licenses may be granted while CE is unreachable. This is the constitutional default: when governance infrastructure is absent, autonomous action stops. The halt must be self-executing — no human intervention is required to trigger it. Recovery from halt is automatic when CE becomes reachable again.
Confidence:      100%
Source:          C-023 (Evidence First — no action returns success without CE confirmation; if CE is unreachable, Evidence First cannot be satisfied, therefore the action cannot proceed); C-001 (Human Override — human override is only possible when the override mechanism (Emergency Stop) is available; CE is the Emergency Stop executor; therefore, CE unavailability means Emergency Stop is also unreachable, which is a constitutional crisis requiring halt); C-003 (Authority licensed — without CE, authority licensing cannot be verified; acting under unverified authority violates C-003); Article IX, Section 3 (Constitutional Floors — platforms must degrade gracefully, never silently drop constitutional guarantees).
Depends On:      C-023 (Evidence First), C-001 (Human Override), C-003 (Authority Licensed Through Evidence)
Produces:        ADR-031 (CE Fail-Safe Behavior on Unavailability); CE circuit breaker in Business Platform and Professional Runtime; CE unavailability detection in health check (grpc_health_probe + timeout 2s); PAAS session halt signal on CE unreachable; constitutional.engine_availability_events table; CCT-CE-AVAIL-01 (chaos test: CE killed → verify all PAAS sessions halt within 5s)
Contradicted By: NONE — a "best effort" mode where actions proceed without CE authorization is a constitutional violation, not a resilience strategy. There is no acceptable degraded mode where actions execute without Evidence First compliance.
Status:          RATIFIED — 2026-07-23 (Founder authorization)
Reviewer Notes:  WHY THERE IS NO DEGRADED MODE FOR CE UNAVAILABILITY:

                 Some platforms implement "best effort" modes where, if a governance service
                 is down, operations continue with reduced oversight. This is unconstitutional
                 for WAOOAW for three reasons:

                 1. EVIDENCE FIRST (C-023): If CE is down, evidence cannot be recorded before action.
                    The action would be constitutionally unauthorized. Proceeding anyway creates
                    an evidence gap — the most fundamental constitutional violation possible.

                 2. AUTHORITY VERIFICATION (C-003): Without CE, the platform cannot verify whether
                    the authority level in the JWT corresponds to a valid, current AuthorityLicense.
                    Acting under potentially revoked authority is a C-003 violation.

                 3. TRUST (C-002): If the platform sometimes bypasses governance when it's
                    inconvenient, then governance is a performance, not a principle.
                    Customers cannot build trust on a platform that only governs when it's easy.

                 WHAT CONSTITUTES CE UNAVAILABILITY:
                 - gRPC health probe fails for > 2 consecutive checks (5s total with 2.5s interval)
                 - Any gRPC call to CE returns UNAVAILABLE status code
                 - CE health endpoint returns NOT_SERVING

                 WHAT HAPPENS ON UNAVAILABILITY:
                 - Business Platform: returns 503 Service Unavailable with Retry-After header for
                   any endpoint that requires CE (employment actions, approval transitions, authority grants)
                 - Professional Runtime: pauses all active PAAS execution loops (Temporal heartbeat
                   stops; no new activities dispatched); active sessions enter PAUSED_CE_UNAVAILABLE state
                 - Web App: Emergency Stop button remains visible but shows "Governance Service Temporarily
                   Unavailable" status; queued stop requests are recorded locally and submitted on recovery
                 - Customer notification: Steward Assistant notified within 60 seconds via WhatsApp

                 WHAT CONTINUES DURING CE UNAVAILABILITY:
                 - Read operations (evidence viewing, approval history) — these do not need CE
                 - Keycloak authentication — independent of CE
                 - Emergency Stop button (WebSocket stays alive; request queued for CE recovery)
                 - Steward interface — read-only mode

                 RECOVERY:
                 - Automatic: When CE health probe returns SERVING, platform resumes within 30s
                 - PAAS sessions: resume from last confirmed evidence state (replay from CE)
                 - Queued Emergency Stops: submitted immediately on recovery (before any other operation)
                 - SLA target: CE recovery within 2 minutes (Azure Container Apps restart SLA)
                 - constitutional.engine_availability_events: records every unavailability event + duration

                 See architecture/reference/graceful-degradation.md for the full degradation manifest
                 and ADR-031 for the implementation specification.

                 MOBILE EXTENSION NOTE (2026-07-23):
                 Mobile clients experience CE unavailability differently from server-side
                 services. A mobile app on a poor cellular connection may intermittently
                 lose CE reachability. The C-079 constitutional position for mobile is:

                 COMPLIANT MOBILE BEHAVIOR when CE is unreachable:
                 - Display visible halt indicator to user ("Actions paused — reconnecting")
                 - Queue READ-ONLY operations locally (viewing, browsing — no CE gate)
                 - REJECT all CE-gated actions with a clear user message
                 - Retry CE connection every 30 seconds with exponential backoff
                 - Resume automatically when CE becomes reachable

                 NON-COMPLIANT (PROHIBITED) mobile behavior:
                 - Executing CE-gated actions offline ("optimistic execution")
                 - Silently queueing CE-gated actions for later replay
                 - Showing "processing" UI while CE is unreachable
                 - Any "degraded mode" that allows actions without Evidence First

                 The distinction: read-only operations (viewing agent status, reading
                 audit records) do not require CE authorization and may proceed offline.
                 Any action that changes state (hire, instruct, stop, approve) requires
                 CE and is prohibited while CE is unreachable.

                 This behavior must be implemented in the mobile client and verified by
                 a CCT before the mobile sprint is authorized to deploy to production.
